# REslava.Result v1.52.0

## ✨ What's New

### 🔬 Pipeline Runtime Observation — Core Library (Phase 1)

Opt-in zero-overhead runtime instrumentation for any `[ResultFlow]` pipeline. No changes to production code required.

**New types in `REslava.Result`:**

- **`ResultFlowObserver`** — abstract base class; override `OnPipelineStart`, `OnPipelineEnd`, `OnNodeEnter`, `OnNodeExit`; netstandard2.0 compatible
- **`RingBufferObserver`** — ready-to-use concrete observer; bounded ring buffer (default 50 executions); thread-safe; `GetTraces()` returns a snapshot
- **`PipelineObserver`** — static registration API: `Register()` / `Unregister()` / `RegisterScoped()` / `AddPipelineObserver<T>()` (DI, net8+)
- **`PipelineTrace`** / **`NodeTrace`** — immutable records carrying per-execution and per-node data (method name, input/output values, error type, elapsed ms, node ID)

**Hooks wired** into `Bind`, `Map`, `Ensure`, `Tap`, `TapOnFailure`, `Match` — zero overhead when no observer registered.

**`{MethodName}_Traced` extension method** — generated by both `REslava.Result.Flow` and `REslava.ResultFlow` for every instance-method `[ResultFlow]` class. Wraps the real method; seeds exact `pipelineId` + `nodeIds`; nested-pipeline safe.

```csharp
var buffer = new RingBufferObserver();
PipelineObserver.Register(buffer);

// One-word change: call _Traced instead of the real method
var result = orderService.PlaceOrder_Traced(request);

foreach (var trace in buffer.GetTraces())
{
    Console.WriteLine($"{trace.MethodName} — {(trace.IsSuccess ? "✓" : "✗")} {trace.ElapsedMs}ms");
    foreach (var node in trace.Nodes)
        Console.WriteLine($"  {node.StepName} → {node.OutputValue}");
}
```

### 🌐 `REslava.Result.Diagnostics` — New Package (Phase 2)

Bridges the `RingBufferObserver` ring buffer to the VS Code VSIX Live panel via a lightweight HTTP endpoint.

```bash
dotnet add package REslava.Result.Diagnostics
```

**`PipelineTraceHost.Start(buffer, port)`** — standalone embedded Kestrel; for console/worker apps:

```csharp
var buffer = new RingBufferObserver();
PipelineObserver.Register(buffer);
using var host = PipelineTraceHost.Start(buffer, port: 5297);
// GET http://localhost:5297/reslava/traces → JSON array of PipelineTrace
```

**`app.MapResultFlowTraces(buffer)`** — for ASP.NET Core apps:

```csharp
var buffer = new RingBufferObserver();
PipelineObserver.Register(buffer);
app.MapResultFlowTraces(buffer); // maps GET /reslava/traces
```

### 🖥️ VSIX v1.3.0 — Live Panel

The VS Code extension now includes a real-time execution debugger for `[ResultFlow]` pipelines.

**How to open:**
- `▶ Debug` CodeLens above any traceable `[ResultFlow]` method
- `$(debug-alt)` inline button in the ⚡ Flow Catalog sidebar
- `🔬 Debug` button in the diagram preview toolbar

**Debug visibility filtering** — all three entry points are shown only for instance-method `[ResultFlow]` classes (those that have `_Traced` generated). Static-method classes hide Debug entirely.

**Live Panel modes:**

| Mode | Behaviour |
|------|-----------|
| **History** | Live scrolling list of all recent executions — method name, outcome (✓/✗), elapsed ms, timestamp; click a row to step through it |
| **Step** | Node-by-node stepper — shows `Node N of M · Input: {value}`; step name, output value, elapsed ms per node; Prev/Next navigation |
| **Replay** | Auto-advances at 800 ms/node; Pause button; loops on completion |

**Diagram node highlight** — the active node is highlighted in amber in the Mermaid diagram, in sync with the stepper. Uses the same FNV-1a hex node IDs as the diagram — no mapping table needed.

### 🐛 Fixed

- **Diagram node highlight off-by-one** — node highlight was always one step ahead; fixed in `Bind`, `Map`, `Match`, `Tap`, `Ensure`
- **`_Traced` node IDs** — generator was emitting source-location strings instead of FNV-1a hashes; now matches Mermaid diagram node IDs exactly
