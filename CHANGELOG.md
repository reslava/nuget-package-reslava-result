## [1.53.0] - 2026-04-05

### ✨ Added

#### FlowProxy — Always-On Tracing + Debug Mode

- **`FlowProxy`** — generated by both `REslava.Result.Flow` and `REslava.ResultFlow` for every `partial` class with `[ResultFlow]` methods; emits a nested `{ClassName}FlowProxy` type accessible via `svc.Flow`; replaces `_Traced` (removed)
- **`svc.Flow.{Method}()`** — always-on proxy call; seeds the pipeline observer on every call; zero overhead when no observer is registered; works for both instance and static patterns
- **`DebugProxy`** — always emitted alongside `FlowProxy`; accessible via `svc.Flow.Debug.{Method}()`; captures a single execution trace and saves it to `reslava-debug-{method}.json` via `RingBufferObserver.Save()`
- **`REF004` analyzer** — diagnostic fires on `[ResultFlow]` methods in a non-`partial` class; code fix adds `partial` to the class declaration; both packages

#### Debug Panel — File-Drop Workflow + Multi-File Picker

- **`RingBufferObserver.Save(path?)`** — saves the current ring buffer to a JSON file; `null` → `reslava-traces.json` in `AppContext.BaseDirectory`; short name (no path separator) → auto-prefixed to `reslava-{name}.json` in `AppContext.BaseDirectory`; full path → exact write (escape hatch); BOM-free UTF-8
- **`DebugProxy` naming** — `DebugProxy` saves to `reslava-debug-{method}.json` (unique per method, prevents overwrite collisions between demo sections)
- **VSIX file watcher** — watches `**/reslava-*.json` (extended from `**/reslava-traces.json`) to catch all method-named debug files
- **Multi-file picker** — when >1 `reslava-*.json` file exists, a `<select>` bar appears in the Debug panel; labels strip the `reslava-` prefix; switching selection reloads the panel from that file
- **Source badge** — panel header shows ⏳ (no data), 📄 (file), or 🌐 (HTTP) to indicate the active data source
- **"Live Panel" → "Debug panel"** — renamed throughout VSIX (command, title, panel header, CodeLens tooltip, sidebar button tooltip); `resultflow.openDebugPanel` command; VSIX v1.4.0

### 🗑️ Removed

- **`_Traced` generated extension** — removed from both packages; replaced by `FlowProxy` / `DebugProxy` which provide a cleaner always-on + debug-mode API without requiring call-site changes

## [1.52.0] - 2026-03-31

### ✨ Added

#### Pipeline Runtime Observation — Phase 1: Core Library

- **`ResultFlowObserver`** — new `abstract class` in `REslava.Result`; `virtual void` methods `OnPipelineStart`, `OnPipelineEnd`, `OnNodeEnter`, `OnNodeExit`; netstandard2.0 compatible; extend without breaking changes
- **`PipelineObserver`** — `public static class`; `Register(ResultFlowObserver)` / `Unregister()` / `RegisterScoped(ResultFlowObserver)` (returns `IDisposable`); `AddPipelineObserver<T>()` IServiceCollection extension (net8+)
- **`RingBufferObserver`** — concrete `ResultFlowObserver`; `ConcurrentQueue<PipelineTrace>`-backed; configurable `Capacity` (default 50); evicts oldest on overflow; `GetTraces()` returns a snapshot; thread-safe
- **`PipelineTrace`** / **`NodeTrace`** — `public sealed record` types; `PipelineTrace` carries `PipelineId`, `MethodName`, `InputValue`, `IsSuccess`, elapsed ms, and `IReadOnlyList<NodeTrace>`; `NodeTrace` carries `NodeId`, `StepName`, `OutputValue`, `ErrorType`, elapsed ms
- **`ResultPipelineHooks`** — `internal static` class; `AsyncLocal<PipelineState?>` for per-execution context (zero cross-pipeline interference); hooks wired into `Bind`, `BindAsync`, `Map`, `MapAsync`, `Ensure`, `EnsureAsync`, `Tap`, `TapAsync`, `TapOnFailure`, `Match`, `MatchAsync`; zero overhead when no observer is registered (null check on hot path)
- **`{MethodName}_Traced` extension** — generated by both `REslava.Result.Flow` (semantic) and `REslava.ResultFlow` (syntax-only) for every instance-method `[ResultFlow]` class; seeds exact `PipelineState` with `pipelineId` + `nodeIds` array; restores previous state on return (nested pipeline safe); call it instead of the real method for exact-tier observation

#### Pipeline Runtime Observation — Phase 2: `REslava.Result.Diagnostics` + VSIX Live Panel

- **`REslava.Result.Diagnostics`** — new NuGet package (`net8+`); bridges the `RingBufferObserver` ring buffer to the VSIX Live panel via a lightweight HTTP endpoint
  - `PipelineTraceHost.Start(buffer, port = 5297)` → `IDisposable` — embedded Kestrel; for console/worker apps that don't have a `WebApplication`
  - `app.MapResultFlowTraces(buffer)` — maps `GET /reslava/traces` into an existing `WebApplication`; overload without `buffer` reads `PipelineObserver.Current as RingBufferObserver` (returns 503 if not registered)
- **VSIX v1.3.0 — Live Panel** — new `ResultFlowLivePanel` webview; polls `GET http://localhost:{port}/reslava/traces` every 2 s
  - **`▶ Debug` CodeLens** — shown only on instance-method `[ResultFlow]` classes (those that have `_Traced` generated); absent on static-method classes
  - **`🔬 Debug` toolbar button** — in the diagram preview webview; hidden when `_Traced` is not generated for the class
  - **Sidebar inline debug button** — `$(debug-alt)` icon on method tree items; shown only for traceable methods
  - **History mode** — live scrolling list of all recent executions; execution number + method name + outcome (✓/✗) + elapsed ms + timestamp; click a row to enter Step mode
  - **Step mode** — node-by-node stepper for a selected trace; shows `Node N of M · Input: {value}`; each step displays step name, output value, elapsed ms; Prev/Next navigation
  - **Replay mode** — auto-advances the stepper at 800 ms per node; pause button; loops back to start on completion
  - **Diagram node highlight** — active node highlighted in amber (`fill:#ffe066`) in the Mermaid diagram; clears on mode exit; `NodeTrace.NodeId` = same FNV-1a hex hash as the Mermaid diagram node ID — no mapping table needed
  - **Debug visibility filtering** — all three Debug entry points (CodeLens, toolbar button, sidebar button) are hidden for static-method `[ResultFlow]` classes; detection: presence of `{ClassName}_Traced_Extensions` in the generated `*_Flows.g.cs` file

### 🐛 Fixed

- **Diagram node highlight off-by-one** — `ConsumeIndex()` (which increments `NodeIndex`) was being called before `CurrentNodeId()` (which reads it), causing the highlight to be one step ahead; fixed in `Bind`, `Map`, `Match`, `Tap`, `Ensure` extension methods
- **`_Traced` node IDs** — the generator was emitting source-location strings (`"Program.cs:42"`) as node IDs instead of FNV-1a hashes; fixed to use `ShortHash.Compute(pipelineId, methodName, index)` — matching the Mermaid diagram node IDs exactly

## [1.51.0] - 2026-03-27

### ✨ Added

#### Pipeline Identity — PipelineId, NodeId & Namespace

- **`pipelineId`** — deterministic 8-hex-char FNV-1a hash of `(fullyQualifiedType, assembly, methodName, paramTypes)` added to every `{MethodName}_Info` JSON in `*_PipelineRegistry.g.cs`; stable across builds and refactors (both packages)
- **`nodeIds`** — ordered JSON array of node identity hashes in `{MethodName}_Info`; each node hash = `ShortHash(pipelineId, type, method, params, index)`; order-aware and collision-safe (both packages)
- **`namespace`** — fully qualified namespace of the containing class added to `{MethodName}_Info` (both packages)
- **`ShortHash`** — new `internal static class ShortHash` in each generator project; FNV-1a 32-bit → 8 lowercase hex chars; deterministic across processes

#### NodeId on Mermaid Diagram Nodes

- **Deterministic node IDs** — Mermaid node IDs in `*_Flows.g.cs` now use `pipelineId`-derived identifiers (e.g. `N0_a1b2c3d4_3f9e`) instead of positional letters (`A`, `B`, `C`); enables runtime → diagram correlation via `ReasonMetadata.NodeId` (both packages)

#### Node SourceLine Fix

- **`MemberAccess.Name` location** — per-node `sourceLine` in `click` directives and registry `_Info` now uses `MemberAccessExpressionSyntax.Name.GetLocation()` instead of the full invocation location; fixes off-by-one on vertical fluent chains where the method name is on a different line than the opening `(` (both packages)

#### `_TypeFlow` Constant

- **`{MethodName}_TypeFlow`** — new generated constant per `[ResultFlow]` method in `*_Flows.g.cs`; same node structure as the main diagram but success edges carry the `Result<T>` type label (e.g. `-->|Result<Order>|` instead of `-->|ok|`); emitted by `ResultFlowMermaidRenderer` when `typeLabels: true` (both packages)

#### VSIX v1.2.1 — Health Icon, Namespace Grouping & Loading Indicator

- **Health icon** — per-method status badge in the Flow Catalog tree derived from registry `_Info`: ✅ has diagram + declared errors · ⚠️ has diagram, no errors · ❌ no diagram or registry missing
- **Namespace grouping** — intermediate namespace nodes (top-two segments) between project root and class list; groups classes by namespace for large-workspace navigation
- **Loading indicator** — project node shows `$(loading~spin)` while `dotnet build` is running; restores to status icon on `onDidEndTaskProcess`

#### `errorTypes` Depth-2 Scan

- **Depth-2 body scan** — when collecting `errorTypes` for registry `_Info` and diagram annotation, the generator now descends into directly-called `Result<T>` methods up to depth 2; catches errors in helpers like `.Bind(o => ValidateOrder(o))` where `ValidateOrder` internally calls `Result.Fail(new XxxError())`; unconditional, independent of `MaxDepth` (both packages)

#### FAIL Node Error Annotation

- **Inline label (1–3 errors)** — FAIL node label now displays error type names inline with `Error` suffix stripped: `FAIL(["fail\nNotFound\nValidation"])` (both packages)
- **Tooltip (4+ errors)** — four or more distinct error types shown as `FAIL(["<span title='NotFound, Validation, ...'>ℹ️fail</span>"])` hover tooltip (both packages)
- **`%% pipelineId:` comment** — `_Diagram` and `_TypeFlow` constants now emit `%% pipelineId: {id}` immediately after `flowchart LR`; enables tooling to correlate a rendered diagram to its registry `_Info` entry (both packages)

#### Legend Update

- **`Legend` constant** — updated `BuildLegendMermaid()` in both packages to include a second note row listing all generated constant types (`_Diagram · _TypeFlow (edge type labels) · _LayerView · _Stats · _ErrorSurface · _ErrorPropagation`) and FAIL annotation formats (`1–3 errors inline · 4+ as ℹ️ tooltip`)

#### VSIX v1.2.2 — Error Children, Layer Tabs, Revised Health & Walkthrough

- **Error children** — method nodes with `errorTypes` now expand to show each error type as an amber `$(warning)` child node; click navigates to the error class definition via workspace symbol search
- **Revised health icon semantics** — ✅ when `errorTypes.length > 0` · ⚠️ when `Bind`/`Gatekeeper` present and `errorTypes` empty · ⚪ `Terminal`/`Map`/`Tap`-only pipeline · ❌ no diagram or registry
- **Layer diagram tabs** — webview panel toolbar adds `_LayerView`, `_Stats`, `_ErrorSurface`, `_ErrorPropagation` tab buttons; tab hidden when constant absent from generated file
- **First-run walkthrough** — VS Code `contributes.walkthroughs` entry with 3 steps: install NuGet package (Track A / Track B with links), add `[ResultFlow]`, open diagram
- **Missing package detection** — `activate()` scans workspace `.csproj` files; if no ResultFlow package found, shows notification with "Install Track A", "Install Track B", "See Demo" buttons; single-project workspaces auto-select `.csproj`; multi-project shows Quick Pick
- **Empty-state tree item** — shown after a completed scan when no pipelines found; tooltip includes Track A/B install instructions and link to `REslava.Result.Flow.Demo`

#### Two-Track Package Messaging

- **Two-track install table** — `README.md`, NuGet READMEs for `REslava.Result`, `REslava.Result.Flow`, `REslava.ResultFlow`, and the VSIX README all now present a Track A / Track B comparison table with explicit `dotnet add package` install blocks

### 🐛 Fixed

#### NuGet Transitive Dependency Fix

- **`REslava.Result.Flow`** — removed `SuppressDependenciesWhenPacking=true`; added `ProjectReference` to `REslava.Result` (non-private); `REslava.Result` now appears as a transitive dependency in the package `.nuspec`
